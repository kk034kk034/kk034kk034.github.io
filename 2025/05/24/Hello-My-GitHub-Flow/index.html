Hello, My GitHub Flow

發表於 2025-05-24 分類於 vcs 閱讀次數：

GitHub Flow 教學：和開源作者使用同一套工作流程

對我來說，GitHub Flow 不只是工具使用的技巧，而是一種「工程習慣」。這篇是我想推薦給每個工程師的版本控制基礎，無論你是獨立開發還是團隊合作，只要掌握這個流程，協作的效率和品質都能大幅提升。

誠心推薦的教學影片：十分钟学会正确的 GitHub 工作流

🧭 為什麼要用 GitHub Flow？

因為「git commit -am ‘fix something’」直接推 main 的日子該結束了。這種做法不僅容易把壞 code 推上去，也讓團隊難以追蹤、回溯與協作。

GitHub Flow 是一套非常實用的流程：

  1. 為每個功能建立分支
  2. 本地提交並推送到 GitHub
  3. 與 main 分支保持同步（rebase）
  4. 發起 Pull Request，進行 code review
  5. 使用 squash and merge 合併
  6. 刪除分支，保持倉庫整潔

🚧 建立功能分支

在 main 上開發是禁忌。先從 main 切出分支再動手，不管是新增功能還是修 Bug，都要獨立開發、獨立測試，這樣出問題才不會拖垮整個專案。

養成 good commit message 的習慣，後面回頭看的時候會感謝自己。建議在提交前用 git diff 看一下，確保不小心加了 debug 或多餘的修改。

🔁 與 main 同步，保持乾淨提交歷史

功能開發不會在真空中進行。main 分支的進度可能已經更新，我們要透過 rebase 整合最新的變更：

rebase 會讓提交記錄保持線性，是 GitHub Flow 推薦的做法。若出現衝突，就手動解、git rebase --continue。

🚀 發起 Pull Request（PR）

當功能開發完成後，推上 GitHub：

進到 GitHub 倉庫，發起 Pull Request，指明要合併到 main。在 PR 裡可以：

  • tag reviewer（協作者）
  • 解釋變更動機與內容
  • 附上截圖或測試方式

🧹 使用 Squash and Merge 合併

PR 通過後，請選擇 Squash and Merge 而不是 Merge commit：

✅ 優點：

  • 合併進 main 時只留下 1 筆 commit，保持簡潔歷史
  • 雜亂的小 commit 不會污染主線

💡 小結：流程讓合作變得更順

這套 GitHub Flow 不一定是唯一解，但對我來說，它解決了很多在協作中常見的狀況，例如：

  • 想回頭看一段功能是什麼時候加的、為什麼改，結果歷史一片「fix bug」、「try again」…
  • 一改就壓到別人的 code，才發現自己忘記先同步 main
  • 做完功能不知道怎麼合併、review 時搞不清楚邏輯變更在哪

透過分支、rebase 和 PR review，把每次變更切得更清楚、有脈絡，也讓自己習慣在合併前檢查、思考與整理。

如果你也曾遇過類似困擾，也許可以試試看這種做法，看看有沒有幫助。
